# QFC AI 推理网络的去中心化模型存储

> 最后更新: 2026-03-08 | 版本 1.0
> GitHub Issue: #5
> 作者: Alex Wei, 产品经理 @ QFC Network

---

## 1. 摘要

QFC 矿工需要下载和存储 AI 模型权重（80MB–140GB）以执行推理任务。目前，QFC 使用 IPFS（Kubo）存储大于 1MB 的推理结果。本报告评估了去中心化存储方案，以解决更大的挑战：在全球矿工网络中分发和持久化 AI 模型权重。

**核心发现：**

- **仅靠 IPFS 不足以**支撑 GB 级别的模型分发——无持久性保证，Pinning 生态正在碎片化（Infura 已弃用，nft.storage 运营困难）
- **Filecoin 冷存储**价格极低（约 $0.002/GB/年），但检索需要解封（45–210 秒）——仅适用于归档
- **Arweave（通过 Irys）**提供一次性约 $0.03/GB 的永久存储——最适合存储标准模型快照
- **Walrus**（Sui）拥有优雅的 Red Stuff 纠删编码，但尚不成熟（主网 2025 年 3 月上线）且与 Sui 强耦合
- **P2P 集群分发**（基于 libp2p 的 BitTorrent 风格）是矿工间模型共享的最实用方案——零成本、自扩展

**建议**：采用**四层混合架构**——链上注册表 → 持久归档（Filecoin/Arweave）→ P2P 集群分发 → 可选 CDN 回退。推理结果继续使用现有的 IPFS 方案。

## 2. 评估的存储方案

### 2.1 Filecoin

**架构**：复制证明（PoRep）+ 时空证明（PoSt）。数据通过慢速编码封装到 32GB 扇区中，然后通过 zk-SNARK 挑战持续证明存储。

| 属性 | 值 |
|----------|-------|
| 冷存储成本 | 约 $0.002/GB/年（$0.20/TB/年） |
| 热存储（Storacha） | 约 $0.20/GB/年（CDN 级检索） |
| 冷检索延迟 | 45 秒（北美），210 秒（非洲）——需要解封 |
| 热检索（Saturn CDN） | 缓存内容亚秒级 |
| 封装时间 | 每 32GB 扇区需数小时（设计如此） |
| FVM | 通过智能合约实现可编程存储交易 |

**2025 年进展**：数据持有证明（PDP）可在未封装数据上实现快速证明——使 Filecoin 转型为可行的热存储平台。

**结论**：归档极佳（每 TB 仅需几美分/年）。冷检索对模型分发来说太慢。Storacha + Saturn CDN 层可用于热模型服务，但增加了复杂性。

### 2.2 Arweave

**架构**：一次付费，永久存储。捐赠机制确保矿工被无限期激励。数据被编织进 blockweave 结构中。

| 属性 | 值 |
|----------|-------|
| 直接 Arweave 成本 | 约 $0.48/GB 一次性（永久） |
| 通过 Irys（原 Bundlr） | 约 $0.03/GB 一次性（通过交易捆绑便宜 16 倍） |
| 10GB 模型成本 | $0.30（Irys）/ $4.80（直接） |
| 100GB 模型成本 | $3.00（Irys）/ $48.00（直接） |
| Irys 吞吐量 | 100K TPS，毫秒级确认 |
| Arweave 主网 | 约 5 TPS，确认需数分钟 |

**注意事项**：AR 代币价格波动直接影响存储成本。不适合频繁更新。Irys 使用订阅模式（不像 Arweave 主网那样真正永久）。

**结论**：最适合存储具有永久溯源性的不可变模型快照。一次性付费模式对标准模型版本很有吸引力。

### 2.3 Walrus（Sui 生态系统）

**架构**：Red Stuff——二维纠删编码，将数据拆分为主/次分片并分布在存储节点上。节点离线时可自我修复。

| 属性 | 值 |
|----------|-------|
| 复制因子 | 4.5 倍（传统复制为 3-10 倍） |
| 编码大小开销 | 约原始数据大小的 5 倍 |
| 最大存储时长 | 2 年（当前上限） |
| 主网 | 2025 年 3 月 |
| 定价 | 动态，以 WAL 计价，尚未大规模基准测试 |

**注意事项**：非常新，生产环境实践有限。5 倍编码大小开销。与 Sui 生态系统紧密耦合。

**结论**：在技术上对大型数据块很优雅，但对当前的 QFC 来说过于不成熟且与 Sui 耦合。值得在 2027 年之后关注。

### 2.4 IPFS + Pinning 服务（QFC 当前方案）

| 服务 | 状态 | 成本 | 备注 |
|---------|--------|------|-------|
| **Pinata** | 活跃 | 约 $20/月，1TB + 500GB 带宽 | 最易用，无密码学保证 |
| **Infura** | 已弃用 | N/A | 2024 年末停止新建 IPFS 密钥 |
| **web3.storage** | 困难 | 约 $0.10/MB 一次性 | 2025 年 4 月公开请求捐款 |
| **Storacha** | 活跃 | 未公开 | Filecoin 支持的热存储，可靠性最高 |

**核心限制**：IPFS 没有内置的持久性或激励层。Pinning 服务"本质上是在 AWS 上运行 IPFS 节点的 Web2 公司。"如果公司关闭，数据就会消失。

**结论**：适用于当前的推理结果（<几 MB），但在没有强大的 Pinning/激励层的情况下，不适用于 GB 级模型权重分发。

### 2.5 0G（ZeroGravity）

**架构**：面向 AI 的 L1，包含 0G Chain（EVM 兼容）、0G Storage、0G DA（声称 50 Gbps 吞吐量）、0G Compute。Aristotle 主网于 2025 年 9 月上线。

**结论**：有趣的 AI 原生定位，但太早且未经验证。性能声明（50 Gbps DA）需要独立验证。值得在 2027 年之后跟踪。

## 3. 成本比较

存储一个 **10GB 模型** **1 年**的成本：

| 方案 | 年度成本 | 检索速度 | 持久性 |
|----------|------------|----------------|-------------|
| Filecoin（冷） | 约 $0.02 | 45–210 秒（解封） | 基于交易 |
| Filecoin（Storacha 热） | 约 $2.40 | 亚秒级（CDN） | Filecoin 支持 |
| Arweave（直接） | $4.80（一次性，永久） | 秒级（网关） | 永久 |
| Arweave（通过 Irys） | $0.30（一次性） | 毫秒级 | Irys 支持 |
| IPFS + Pinata | 约 $2.40 | 秒级 | 依赖服务商 |
| AWS S3（参考） | $27.60 | 毫秒级 | AWS SLA |

对于 **100GB 模型**——按比例乘算。Filecoin 冷存储：约 $0.20/年。Arweave 通过 Irys：约 $3.00 一次性。

## 4. 推荐架构：四层混合

### 4.1 概览

```
┌─────────────────────────────────────────────────────┐
│  第一层：链上模型注册表                                │
│  - 模型元数据（哈希、版本、大小、格式）                  │
│  - 通过 CID 进行内容寻址                              │
│  - 治理：模型审批/下线                                 │
├─────────────────────────────────────────────────────┤
│  第二层：持久归档存储                                  │
│  - Filecoin 冷存储（$0.002/GB/年）                    │
│  - 或 Arweave/Irys（$0.03/GB 一次性）                │
│  - 标准版本的真实数据源备份                             │
├─────────────────────────────────────────────────────┤
│  第三层：P2P 集群分发（主要）                           │
│  - 基于 libp2p 的模型分发守护进程                      │
│  - 每个矿工 = 做种者 + 下载者                         │
│  - 分块传输（4GB 分块，并行下载）                      │
│  - 自扩展：矿工越多 = 分发越快                         │
├─────────────────────────────────────────────────────┤
│  第四层：引导 / CDN 回退                               │
│  - 3-5 个基金会种子节点（美东、欧洲、亚太）              │
│  - Storacha/Saturn CDN 用于冷启动区域                  │
│  - 仅在集群密度低时使用                                 │
└─────────────────────────────────────────────────────┘
```

### 4.2 第一层：链上模型注册表

```move
resource ModelRegistration {
    id: UID,
    model_id: String,           // "qfc-llm-7b"
    version: String,            // "v1.2.0"
    content_hash: Hash,         // 模型权重的 SHA-256
    cid: String,                // IPFS 兼容的内容标识符
    size_bytes: u64,            // 模型总大小
    format: String,             // "onnx", "gguf", "safetensors"
    quantization: String,       // "fp16", "q4_k_m", "awq"
    archival_cid: Option<String>, // Filecoin/Arweave 引用
    approved: bool,             // 治理已批准
    registered_at: u64,
}
```

**完整性验证**：矿工对照链上 `content_hash` 验证已下载的模型权重。任何哈希不匹配 → 拒绝并重新下载。

### 4.3 第二层：持久归档

对每个治理批准的模型版本：

1. 通过 FVM 存储交易归档到 **Filecoin 冷存储**（约 $0.002/GB/年）
2. 可选归档到 **Arweave（通过 Irys）**作为永久备份（约 $0.03/GB）
3. 将归档 CID/txID 存储在链上 `ModelRegistration` 中

**QFC 初始 20 个模型注册表的总成本**（预估总计约 500GB）：
- Filecoin：约 $1.00/年
- Arweave/Irys：约 $15.00 一次性

### 4.4 第三层：P2P 集群分发

主要分发机制。每个已下载模型的矿工都成为做种者。

**协议设计**（基于 libp2p，类似 BitTorrent）：

```
新矿工加入 QFC 网络：
  1. 查询链上 ModelRegistry 获取所需模型
  2. 对于每个模型：
     a. 发现拥有该模型的节点（libp2p DHT / PubSub 公告）
     b. 从多个节点并行下载分块（4GB 分块）
     c. 对照清单验证每个分块的哈希
     d. 对照链上 content_hash 验证完整模型哈希
     e. 开始向其他节点做种
```

**为什么可行**：
- **零边际成本**：矿工共享其已有的带宽
- **自扩展**：网络随增长变快（更多做种者）
- **已兼容**：QFC 使用 libp2p（通过 Kubo/IPFS）——集群化是自然扩展
- **成熟模式**：Petals、GOAT 和 Hugging Face 都用这种方式分发 LLM 权重

**带宽激励**：做种更多模型数据的矿工获得少量 PoC 网络评分加成（符合现有 10% 网络维度）。

### 4.5 第四层：引导回退

针对冷启动场景（新区域、冷门模型）：

- **3-5 个基金会运营的种子节点**在关键区域（美东、西欧、亚太）
- 每个种子节点维护所有治理批准的模型
- Storacha/Saturn CDN 作为额外回退
- 种子节点是临时的——一旦某区域有足够矿工，矿工就成为主要来源

## 5. 模型版本管理与生命周期

### 5.1 版本锁定

```
模型版本生命周期：
  1. 提议 → 治理投票
  2. 批准 → 链上注册、归档、种子部署
  3. 活跃 → 矿工下载并提供服务
  4. 弃用 → 新版本可用，30 天迁移窗口
  5. 下线 → 从注册表移除，矿工可回收存储
```

### 5.2 增量更新

对于模型更新（微调、量化格式变更）：
- **增量分发**：仅分发变更的层/权重，而非完整模型
- 减少常见操作的带宽消耗（例如量化格式变更）
- 仍对照链上注册表验证完整模型哈希

### 5.3 每个矿工等级的存储需求

| 等级 | 所需模型 | 预估存储 |
|------|----------------|-------------------|
| 冷 | 仅嵌入模型 | 约 200MB |
| 暖 | 嵌入 + 分类 + 小型 LLM | 约 1.5GB |
| 热 | 所有标准模型（含 7B LLM） | 约 15GB |
| 超级 | 所有模型（含 70B+） | 约 150GB |
| 安全（TEE） | TEE 兼容模型 | 视情况而定 |

| 功能 | Bittensor | Ritual | io.net | Akash | **QFC（目标）** |
|---------|-----------|--------|--------|-------|-----------------|
| 模型存储 | 无（链下） | Docker 镜像 | 无 | Docker 镜像 | **链上注册表 + P2P 集群** |
| 模型验证 | 无 | 容器哈希 | 无 | 容器哈希 | **链上内容哈希** |
| 分发 | 手动 | Docker Hub | 手动 | Docker 注册表 | **libp2p P2P 集群** |

| 版本管理 | 无 | Docker 标签 | 无 | Docker 标签 | **链上治理** |

**QFC 的优势**：端到端可验证的模型供应链——从链上注册到内容寻址分发再到哈希验证加载。

## 8. 关键设计决策

### 8.1 为什么不用单一存储方案？

没有单一的去中心化存储平台能满足所有需求：
- **Filecoin**：归档便宜，检索慢
- **Arweave**：永久但频繁更新成本高
- **IPFS**：无持久性保证
- **Walrus/0G**：过于不成熟

混合方案让每个方案做其最擅长的事。

### 8.2 为什么选 P2P 集群而非 CDN？

| 属性 | P2P 集群 | 中心化 CDN |
|----------|-----------|----------------|
| 成本 | 免费（矿工共享带宽） | $0.01-0.10/GB 传输 |
| 扩展性 | 随网络增长自扩展 | 成本线性增长 |
| 去中心化 | 完全去中心化 | 单点故障 |
| 冷启动 | 慢（需要初始种子） | 快 |
| QFC 兼容性 | libp2p 已在技术栈中 | 需额外基础设施 |

P2P 是主要路径；CDN 仅作为冷启动回退。

### 8.3 为什么选择内容寻址？

内容寻址存储（CID/哈希）提供：
- **完整性**：任何节点都可以验证数据是否匹配预期哈希
- **去重**：相同模型无论位置如何只存储一次
- **不可变性**：模型版本可永久引用
- **兼容性**：CID 在 IPFS、Filecoin 和 Arweave 生态系统中通用
